# ChatSpend - 自然语言记账Agent应用技术方案

## 一、项目概述

### 1.1 产品定位
个人智能记账助手，支持自然语言交互、多模态输入（文本/语音/图片）、智能分析、主动提醒。

### 1.2 核心价值
- **零学习成本**：自然语言记账，无需手动分类
- **多模态输入**：支持文本、语音、图片识别
- **智能洞察**：自动分析消费习惯，提供理财建议
- **主动服务**：定期汇报、异常提醒、预算管理

### 1.3 技术亮点
- 基于DDD领域驱动设计
- 多智能体架构（Multi-Agent）
- Plan-and-Execute执行框架
- Smart Agent, Dumb Tools设计理念

## 三、多智能体设计

### 3.1 Agent角色定义

#### 3.1.1 编排器 Agent (Orchestrator)
**职责**: 总调度，意图识别，跨Agent协调

**Prompt 核心要素**:
```
你是记账助手的总协调者。你的职责是：
1. 理解用户意图：记账、查询、分析、预算设置、提醒设置？
2. 路由到对应的专项Agent
3. 协调多个Agent完成复杂任务
4. 维护会话上下文，支持多轮对话

工作流程：
- 接收用户输入 → 意图分类 → 生成执行计划 → 分发任务 → 汇总结果 → 返回用户

示例：
用户: "这个月餐饮花了多少钱？"
意图: 查询分析
计划: 
  1. 查询本月餐饮交易
  2. 计算总金额
  3. 对比上月数据
  4. 生成分析报告
分发: AnalysisAgent.execute(plan)
```

#### 3.1.2 记账 Agent (Recording Agent)
**职责**: 解析自然语言，处理多模态输入，创建交易记录

**Prompt 核心要素**:
```
你是专业的记账员。你的职责是：
1. 从自然语言中提取交易信息：金额、类别、时间、描述
2. 处理模糊表达："昨天"、"前两天"、"中午"
3. 智能分类：根据描述推断消费类别
4. 处理多模态输入：文本、语音、图片

关键能力：
- 时间推理：今天=2026-03-30，昨天=2026-03-29，前天=2026-03-28
- 金额标准化："35块" → 35.00，"一百二" → 120.00
- 分类推理："打车" → 交通出行，"午饭" → 餐饮美食
- 容错处理：信息缺失时主动询问

示例：
输入: "今天午饭花了35块"
输出: {
  "amount": 35.00,
  "category": "餐饮美食",
  "subcategory": "午餐",
  "date": "2026-03-30",
  "description": "午饭",
  "confidence": 0.95
}
```

#### 3.1.3 分析 Agent (Analysis Agent)
**职责**: 消费分析、趋势预测、报表生成、建议生成

**Prompt 核心要素**:
```
你是财务分析师。你的职责是：
1. 分析消费模式：类别分布、时间规律、异常消费
2. 预测未来趋势：基于历史数据预测消费
3. 生成可视化报表：图表、趋势线、对比分析
4. 提供理财建议：基于消费习惯给出优化建议

分析维度：
- 时间维度：日/周/月/年
- 类别维度：餐饮、交通、娱乐、购物...
- 趋势维度：环比、同比、移动平均
- 异常维度：突发大额、超预算、异常频率

输出格式：
- 数据表格：清晰的数字统计
- 可视化图表：饼图、柱状图、折线图
- 文字洞察：关键发现和建议
```

#### 3.1.4 预算 Agent (Budget Agent)
**职责**: 预算设置、进度跟踪、超支警告、调整建议

**Prompt 核心要素**:
```
你是预算管理专家。你的职责是：
1. 设置预算：总预算、分类预算、周期预算
2. 实时跟踪：已用金额、剩余额度、使用率
3. 超支预警：接近上限提醒、超支警告
4. 调整建议：基于实际情况建议预算调整

预算类型：
- 月度总预算：每月总支出上限
- 分类预算：餐饮、交通、娱乐等分项预算
- 专项预算：旅行、购物等特定事项预算

预警机制：
- 50%: 温馨提示
- 80%: 注意提醒
- 100%: 超支警告
- 120%: 严重超支警告
```

#### 3.1.5 提醒 Agent (Reminder Agent)
**职责**: 定期汇报、异常提醒、理财建议、目标跟踪

**Prompt 核心要素**:
```
你是贴心的财务管家。你的职责是：
1. 定期汇报：每日/周/月消费总结
2. 异常提醒：突发大额、连续消费、预算告急
3. 理财建议：基于消费习惯的优化建议
4. 目标跟踪：储蓄目标进度、预算执行率

提醒类型：
- 时间触发：每天晚上8点汇报今日消费
- 事件触发：单笔消费超过1000元立即提醒
- 条件触发：月预算使用超过80%时警告
- 主动推送：每周一早上发送上周消费报告

个性化：
- 学习用户习惯，调整提醒频率
- 根据用户反馈优化建议质量
```

## 六、Plan-and-Execute框架实现

### 6.1 框架流程

```python
class PlanAndExecuteFramework:
    """
    Plan-and-Execute框架实现
    
    流程：
    1. 接收用户输入
    2. 意图识别 -> 生成计划（Plan）
    3. 用户确认计划（可选）
    4. 批量执行计划步骤（Execute）
    5. 返回结果
    """
    
    async def process(self, user_input: str, context: dict):
        # 1. 意图识别
        intent = await self.intent_recognizer.recognize(user_input, context)
        
        # 2. 生成执行计划
        plan = await self.planner.create_plan(intent, context)
        
        # 3. 执行计划
        results = []
        for step in plan.steps:
            result = await self.executor.execute_step(step)
            results.append(result)
            
            # 如果某步失败，根据策略决定是否继续
            if not result.success and step.critical:
                break
        
        # 4. 汇总结果
        final_result = await self.result_aggregator.aggregate(results)
        
        return final_result
```

#### 10.2.2 离线优先策略
```typescript
// 离线数据同步服务
class OfflineSyncService {
  private syncQueue: Transaction[] = [];
  
  async addTransaction(transaction: Transaction) {
    // 1. 立即保存到本地SQLite
    await localDB.insertTransaction(transaction);
    
    // 2. 添加到同步队列
    this.syncQueue.push(transaction);
    
    // 3. 尝试同步到云端
    if (await this.isOnline()) {
      await this.syncToCloud(transaction);
    }
  }
  
  async syncToCloud(transaction: Transaction) {
    try {
      await api.createTransaction(transaction);
      // 同步成功，从队列移除
      this.syncQueue = this.syncQueue.filter(t => t.id !== transaction.id);
    } catch (error) {
      // 同步失败，保留在队列中，下次重试
      console.log('Sync failed, will retry later');
    }
  }
  
  // 应用启动时自动同步离线数据
  async syncAllPending() {
    for (const transaction of this.syncQueue) {
      await this.syncToCloud(transaction);
    }
  }
}
```

```typescript
class DataSyncManager {

  resolveConflict(localData: Transaction, remoteData: Transaction): Transaction {
    return new Date(localData.updated_at) > new Date(remoteData.updated_at) 
      ? localData 
      : remoteData;
```

```typescript
import * as SecureStore from 'expo-secure-store';

class SecureStorage {
  // 存储加密数据
  async saveSecureData(key: string, value: string) {
    const encryptedValue = await this.encrypt(value);
    await SecureStore.setItemAsync(key, encryptedValue);
  }
  
  // 读取加密数据
  async getSecureData(key: string): Promise<string | null> {
    const encryptedValue = await SecureStore.getItemAsync(key);
    if (encryptedValue) {
      return await this.decrypt(encryptedValue);
    }
    return null;
  }
  
  // 加密函数
  private async encrypt(text: string): Promise<string> {
    // 使用AES加密
    return await Crypto.digestStringAsync(
      Crypto.CryptoDigestAlgorithm.SHA256,
      text
    );
  }
}
```

## 十一、关键技术难点

### 11.1 自然语言理解
**难点**: 模糊时间、金额、类别的准确提取
**方案**: 
- 规则引擎 + LLM混合推理
- 上下文感知（记住上次提到的日期）
- 置信度机制，低置信度时主动确认

### 11.2 多轮对话状态管理
**难点**: 跨轮次的上下文维护
**方案**:
- 会话状态机
- 向量数据库存储历史对话
- LangChain的ConversationChain

### 11.3 智能分类准确性
**难点**: 新兴消费场景、自定义分类
**方案**:
- 可学习的分类规则库
- 用户反馈机制优化分类
- Few-shot learning快速适应新场景

### 11.4 性能优化
**难点**: 复杂分析查询的性能
**方案**:
- 预聚合统计数据
- 异步任务处理
- 缓存热点数据

## 十二、风险与对策

| 风险 | 影响 | 概率 | 对策 |
|------|------|------|------|
| LLM成本过高 | 高 | 中 | 使用开源模型、批量处理、缓存结果 |
| 分类准确率低 | 高 | 中 | 混合推理、用户反馈优化、人工标注数据 |
| 性能瓶颈 | 中 | 低 | 预聚合、异步处理、分布式部署 |
| 数据隐私 | 高 | 低 | 本地优先、加密存储、权限控制 |

## 十三、总结

本技术方案基于王自如的架构理念，采用了：
1. **DDD领域驱动设计** - 清晰的业务边界，独立的领域模型
2. **多智能体架构** - 分工明确的Agent角色，可独立演进
3. **Plan-and-Execute框架** - 提升执行效率和用户体验
4. **Smart Agent, Dumb Tools** - AI负责推理决策，工具只做执行

这套架构具备良好的扩展性、可维护性，能够支撑产品的长期演进。
